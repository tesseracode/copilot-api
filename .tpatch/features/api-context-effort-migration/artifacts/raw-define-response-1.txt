# Acceptance Criteria

1. Model mapping no longer normalizes max/high/xhigh suffixes to -1m variants for Claude 4.6+
2. Effort parameter is extracted from request body and forwarded via `output_config.effort` instead of model name suffix
3. `supportsEffort()` function checks advertised `reasoning_effort` capabilities array instead of -1m suffix presence in model ID
4. All Claude 4.6+ models correctly report `max_context_window_tokens: 1000000` without suffix variants in model list
5. Claude 4.5 and earlier models continue to use `thinking: {type: "enabled", budget_tokens: N}` approach (no reasoning_effort capability)
6. Effort values are validated against model's supported `reasoning_effort` array before forwarding; unsupported values are rejected with clear error
7. [1m] suffix in incoming requests is stripped as no-op (backward compat for Claude Code) but does not trigger variant resolution
8. GPT-5.x effort is forwarded via `reasoning: {effort: "..."}` in responses payload
9. All existing tests pass; new tests added for effort forwarding via `output_config.effort` and validation against capabilities
10. No silent dropping of effort values for opus-4.7+ models; invalid effort values produce HTTP 400 with supported values listed
